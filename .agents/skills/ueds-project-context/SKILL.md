# UEDS Project Context

## Goal

Keep every decision aligned with the official UEDS demonstration and prevent invented or contradictory project claims.

## Instructions

1. Read `docs/PROJECT_CONTEXT.md` before making product or content decisions.
2. Read `docs/FUNCTIONAL_SPEC.md` when planning routes or features.
3. Treat those files as the source of truth.
4. Clearly separate confirmed prototype facts, demo functionality and future ideas.
5. Write all user-facing copy in natural Spanish.
6. Keep the experience mobile-first.
7. When a request conflicts with the documents, explain the conflict before modifying the source of truth.

## Constraints

- Do not invent extraction percentages, certifications or industrial capacity.
- Do not portray the rewards system as a permanent commercial program.
- Do not use lorem ipsum.
- Do not silently change points, rewards or redemption rules.
- Do not redesign the logo concept.

## Example

User request: `Create the homepage hero.`

Expected behavior: Read the context and design files, use the official name, explain the project without false metrics, and include a CTA to explore or register.
